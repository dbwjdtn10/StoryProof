I. 프로젝트 개요 및 아키텍처
================================================================================

[핵심 목적]
소설 텍스트를 자동으로 분석하여 등장인물, 사건, 장소 등을 구조화하고,
RAG(Retrieval-Augmented Generation) 기반 챗봇으로 질의응답 제공

[기술 스택]

Backend:
- FastAPI (Python 3.10+)
- PostgreSQL 15+ (메타데이터)
- Pinecone (벡터 DB)
- Google Gemini 2.5 Flash (LLM)
- BAAI/bge-m3 (임베딩 모델, 1024차원)
- SQLAlchemy + Alembic

Frontend:
- React 18 + TypeScript
- Vite 6.3.5
- Radix UI
- Tailwind CSS (추정)
- Axios

Infrastructure:
- Docker (PostgreSQL)
- Background Tasks: asyncio.to_thread (현재)

[주요 데이터 흐름]

1. 파일 업로드
   → 인코딩 자동 감지 (UTF-8/CP949/EUC-KR)
   → Chapter 레코드 생성 (status=PENDING)

2. 백그라운드 분석
   → DocumentLoader: 파일 로드
   → SceneChunker: 씬 분할 (동적 임계값)
   → GeminiStructurer: LLM 구조화 (인물/장소/사건)
   → EmbeddingEngine: 벡터 생성 및 Pinecone 업로드
   → Analysis 테이블 저장

3. RAG 챗봇
   → 질문 임베딩
   → Pinecone 검색 (코사인 유사도)
   → 상위 5개 씬 컨텍스트 구성
   → Gemini API 답변 생성
   → ChatHistory 저장

III. 진행 중인 기능 🚧
================================================================================

[1] 캐릭터 관계도 시각화 (계획 완료, 구현 대기)
───────────────────────────────────────────────
상태: ✅ 최종 계획 수립 완료 → 구현 시작 가능

핵심 전략:
- Phase 1 (MVP): 체크박스 선택 → 스냅샷 네트워크
- 그래프 라이브러리: Cytoscape.js
- 비용 최적화: 공동 등장 빈도 → 상위 N개만 LLM 분석

2단 관계 라벨 체계 (7개 상위 카테고리):
1. 친밀/우호 (파랑) - 친구, 동료, 신뢰, 보호자
2. 로맨스/애정 (빨강) - 연인, 짝사랑, 배신한 연인
3. 가족/혈연 (초록) - 부모-자식, 형제, 혼인
4. 권력/지배 (보라) - 상사-부하, 스승-제자
5. 갈등/적대 (검정) - 경쟁자, 원수, 복수, 살의
6. 거래/이해 (주황) - 동업, 계약, 동맹
7. 모호/미정 (회색) - 관계 불명

UX 플로우:
1. 인물 헤더 [관계도] 버튼 클릭
2. [보기] 버튼 생성 (비활성) + 체크박스 표시
3. 2개 이상 선택 → [보기] 활성화 (파란색)
4. [보기] 클릭 → 관계도 그래프 표시

시각화 규칙:
- 선 굵기: 관계 강도 (공동 등장 빈도)
- 선 색상: 관계 유형 (7가지 카테고리)
- 선 스타일: 실선(확실) / 점선(추정)

API:
POST /novels/{novel_id}/chapters/{chapter_id}/relationships
Request: { "character_names": ["홍길동", "임꺽정"] }
Response: { "nodes": [...], "edges": [...] }

예상 소요 시간: 8-10시간
상세 문서: relationship_settings.txt
          relationship_visualization_plan.md

Phase 2 (추후):
- 구간 스냅샷 (시작/중간/끝 3탭)
- 타임라인 리본형 (관계 변화 시각화)
- 이벤트 마커 (살해/배신/키스 등)

VI. 기술 부채 및 개선사항 ⚠️
================================================================================

[🔴 Critical (즉시 수정 필요)]

1. 인증 보안 강화
   - JWT Secret Key 기본값 → 강력한 랜덤 키 생성
   - Refresh Token 로직 미구현
   - HTTPS 미적용 (프로덕션 시 필요)

2. 에러 핸들링 개선
   - 백그라운드 작업 실패 시 복구 메커니즘 없음
   - Gemini API 실패 시 부분적 처리만
   - 사용자에게 에러 상세 노출 위험
   권장: Sentry/Rollbar 연동, 구조화된 로깅

3. Database 연결 풀 관리
   - SessionLocal() 직접 호출로 연결 누수 위험
   - context manager 패턴 사용 권장

[🟡 Important (단기 개선)]

4. ⏸️ Celery 도입 (보류 중)
   - 현재: asyncio.to_thread (작업 손실 가능성)
   - 장점: 재시도 정책, 우선순위, 분산 처리
   - 대안:  FastAPI BackgroundTasks (임시)

5. Redis 캐싱 전략
   - 동일 질문 매번 Pinecone 검색
   - Gemini API 비용 증가
   권장: Redis 캐시 (1시간 TTL)

6. API Rate Limiting
   - 무제한 API 호출 가능 (DDoS 취약)
   권장: SlowAPI 도입 (분당 10회 제한)

[🟢 Nice to Have (중장기)]

7. 벡터 검색 고도화
   - Hybrid Search (BM25 + Semantic)
   - Elasticsearch + Pinecone 결합

8. LangChain 통합
   - Conversation Memory
   - Agent 기반 작업 분할

9. 관계도 캐싱
   - 관계 분석 결과 PostgreSQL 저장
   - 필요 시 Redis 캐시
