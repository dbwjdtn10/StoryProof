# ISMS-P 인증 준비 갭 분석 (초안)

> **주의**: 이 문서는 StoryProof가 ISMS-P 인증을 **보유하고 있다는 뜻이 아닙니다.**
> ISMS-P(정보보호 및 개인정보보호 관리체계) 인증은 한국인터넷진흥원(KISA)이
> 지정한 인증기관의 공식 심사를 통해서만 취득 가능하며, 코드 변경만으로는
> 획득할 수 없습니다(조직 정책, 물리보안, 인력, 문서화된 절차, 정기 심사가
> 필요). 이 문서는 대형 서점/플랫폼과의 계약 협상 시 "우리가 지금 뭘 갖췄고
> 뭐가 부족한지" 빠르게 파악하기 위한 **갭 분석 체크리스트**입니다.

## 2. 보호대책 요구사항 — 이미 코드로 구현된 것

| 통제 영역 | 구현 내용 | 근거 |
|---|---|---|
| 인증 | JWT 기반 세션, bcrypt/argon2 비밀번호 해시 | `backend/core/security.py` |
| 접근 통제 | 소유권 기반 테넌트 격리(파트너/이용자는 자기 데이터만 접근) | `backend/core/partner_auth.py`, 각 서비스의 소유권 검증 |
| 접근 통제 | 관리자 전용 API 별도 권한 검증(`require_admin`) | `backend/api/v1/endpoints/partner_admin.py` |
| API 인증 | 파트너 API 키 SHA-256 해시 저장(원본 미보관) | `backend/core/partner_auth.py` |
| 데이터 무결성 | 웹훅 페이로드 HMAC-SHA256 서명 | `backend/services/webhook_service.py` |
| 가용성 | Redis 장애 시 서킷 브레이커(전면 장애 확산 방지) | `backend/core/partner_auth.py`, `docs/SLA.md` §3 |
| 가용성 | DB 커넥션 풀 상향(동시성 대응) | `docker-compose.yml` DB_POOL_SIZE |
| 설정 보안 | 프로덕션 SECRET_KEY 기본값 사용 시 기동 차단 가드 | `backend/core/config.py` |
| 레이트 리밋 | IP/API키 기반 분당 요청 제한 | `backend/api/v1/endpoints/auth.py`, `partner_auth.py` |
| 데이터 최소화 | 콘텐츠 보안 계약(minimal retention) — 원문 삭제 옵션 | `Partner.content_retention_mode`, 2026-07-13 |
| 사용량 추적 | 파트너별 API 호출 로그(과금/이상탐지 근거) | `ApiUsageLog` |
| 개인정보 파기 | 회원 탈퇴 시 계정 삭제 API | `DELETE /api/v1/auth/me` |

## 4. 파트너 계약 협상 시 참고

교보문고/알라딘/카카오페이지 등 대형 파트너는 계약 조건으로 ISMS-P
인증서를 요구할 수 있습니다. 인증 취득 전까지는:
- 위 §2(구현됨) 표를 기술 실사(technical due diligence) 자료로 제공
- §3(부족함) 항목 중 **저장/전송 암호화, 백업**은 실제 클라우드 배포 시
  우선 적용 권장 (파일럿 전 최소 요건)
- 정식 ISMS-P 인증은 별도 예산·일정으로 진행 (통상 수개월 소요, 외부
  컨설팅 필요)
